git remote add origin url_ : used to connect git and github


git remote add origin https://github.com/numpy1234/learning-linux  : You are connecting a brand-new local project to GitHub


git remote set-url origin git@github.com:numpy1234/learning-linux.git  : You already have a remote called origin and want to change it

git remote add origin git@github.com:numpy1234/new-project.git : You want to connect to a different GitHub repository

You don't strictly need to change origin. We changed it because your current origin uses HTTPS, and you were having an authentication problem.

Your current setup:

git remote -v

showed:

origin https://github.com/numpy1234/learning-linux

That means Git connects like this:

Your computer
      |
      | HTTPS
      |
GitHub
      |
 asks for username + token

When you did:

git push origin master

Git asked:

Username for 'https://github.com':
Password:

and failed because GitHub no longer accepts normal passwords.

Changing origin:

git remote set-url origin git@github.com:numpy1234/learning-linux.git

changes only how Git connects:

Before:

origin → https://github.com/numpy1234/learning-linux
          (HTTPS)
          username + token

After:

origin → git@github.com:numpy1234/learning-linux.git
          (SSH)
          SSH key

The repository itself does not change. Your code does not change. Your commits do not change.

You have two choices:

Choice 1: Keep HTTPS

Then don't change origin.

Use a GitHub Personal Access Token when Git asks for a password.

Your remote stays:

https://github.com/numpy1234/learning-linux.git
Choice 2: Use SSH (what we were setting up)

Change origin once:

git remote set-url origin git@github.com:numpy1234/learning-linux.git

Then future pushes work without typing credentials.

So the answer is:

We change origin because your existing remote points to HTTPS, but we want to switch the connection method to SSH. If you prefer using HTTPS with a token, you don't need to change it.
